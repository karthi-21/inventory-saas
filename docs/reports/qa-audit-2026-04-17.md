## 2. Full User Journey Breakdown

### Step 1: Landing Page → Signup
**Expected**: User sees marketing page, clicks "Get Started" or pricing CTA, lands on `/signup?plan=grow`
**Actual**: Works. Landing page has proper CTAs linking to `/signup` and `/login`.

**Issues**:
- Plan data is **triplicated** (signup page, payment page, `plan-limits.ts`) — drift risk
- Auth provider feature flags (`NEXT_PUBLIC_ENABLE_EMAIL_AUTH`, `NEXT_PUBLIC_ENABLE_GOOGLE_AUTH`, `NEXT_PUBLIC_ENABLE_PHONE_AUTH`) are checked at runtime but **not set in `.env.local`** — if these are undefined, NO auth method renders, making signup impossible

### Step 2: Signup → Email Verification
**Expected**: User signs up with email, gets confirmation link, clicks it, returns to `/auth/callback`
**Actual**: Multiple failure modes:

**Issues**:
- `emailRedirectTo` uses `window.location.origin` — correct on client-side
- But the **server-side GET handler** in `/api/auth/callback-server` uses `process.env.NEXT_PUBLIC_SITE_URL` — **this env var does NOT exist** in `.env.local` (which uses `NEXT_PUBLIC_APP_URL` instead)
- Result: OAuth redirects (Google sign-in) will always fall back to `http://localhost:3003`, breaking in production
- Email confirmation links from Supabase will redirect to the Supabase-configured URL (set in Supabase dashboard), not necessarily matching the app URL

### Step 3: Auth Callback → Payment
**Expected**: After email/OAuth verification, user is redirected to `/payment?plan=X`
**Actual**: The callback handler (`/auth/callback`) does:
1. Exchange OAuth code or process email token
2. POST to `/api/auth/callback-server` to create user/tenant in DB
3. Redirect to `/payment?plan=X`

**Issues**:
- The callback creates a **second tenant** if the user already has one (the `findFirst` check may fail if the email changed between calls or if there's a race condition)
- The trial subscription is created as `PRO` plan with **14-day** trial in callback-server, but **30-day** trial in the onboarding route — inconsistent trial periods
- If the POST to `/api/auth/callback-server` fails, user sees "Authentication Error" and is redirected to `/signup` after 3 seconds — no retry mechanism
- On slow connections, the 3-second redirect timer starts before the user can read the error

### Step 4: Payment → Subscription
**Expected**: User selects plan, pays via Dodo Payments, gets redirected to `/payment/success`
**Actual**: Dodo Payments creates a checkout session and redirects the user.

**Issues**:
- Dodo client defaults to **sandbox URL** (`https://api.sandbox.dodopayments.com`) — production must set `DODO_BASE_URL`
- No `DODO_LAUNCH_PLAN_ID` or `DODO_GROW_PLAN_ID` values were found in `.env.local.example` — these are Dodo product IDs that must be created in Dodo dashboard first
- Payment success page tries to verify payment with `sessionId` from URL params, but Dodo redirects with different param names — potential param mismatch
- Payment page shows "Pay ₹999 Now" but user is already on 14-day trial — confusing UX (are they paying now or after trial?)
- Scale plan triggers `mailto:sales@ezvento.karth-21.com` — hard-coded email domain

### Step 5: Payment Success → Onboarding
**Expected**: After payment, auto-redirect to `/onboarding` after 3 seconds
**Actual**: Works but has a **critical route conflict**

**Issues**:
- **TWO competing onboarding pages** exist at the same route `/onboarding`:
  - `src/app/(onboarding)/page.tsx` — 4-step wizard (Business, Store Type, Location, Team) — calls `POST /api/onboarding`
  - `src/app/onboarding/page.tsx` — 2-step wizard (Store Details, Add More Stores) — calls `POST /api/onboarding/create-store`
  - These create **completely different outcomes**: the 4-step wizard creates a full tenant with categories, personas, and settings; the 2-step one only creates a store under an existing tenant
  - Next.js resolves this silently — the build only shows one `/onboarding` route, but which one wins is **undefined behavior**

### Step 6: Onboarding → Dashboard
**Expected**: After store setup, user is redirected to `/dashboard`
**Actual**: Dashboard layout checks for existing stores and subscription status.

**Issues**:
- Dashboard layout redirects to `/onboarding` if no store exists
- Dashboard layout redirects to `/payment` if trial expired or no active subscription
- But `/onboarding` is also auth-protected in middleware — creates a **redirect loop** if the user's session is invalid
- If the wrong onboarding page renders (the 2-step one), it calls `POST /api/onboarding/create-store` which expects an existing tenant — but for a brand new user, the tenant might not have proper settings/personas/categories

### Step 7: Dashboard → POS
**Expected**: User can access `/dashboard/billing/new` for POS
**Actual**: POS page has full feature set including offline, thermal printing, keyboard shortcuts.

**Issues**:
- POS offline sync via IndexedDB/Dexie — untested with actual network drops
- Hardcoded fallback VPA `ezvento@ybl` shown to customers — should use store's configured VPA
- PhonePe integration defaults to preprod sandbox — will fail against production PhonePe API

## 4. Root Cause Analysis

### RC1: Environment Variable Naming Chaos
**Root**: Multiple env var names for the same concept:
- `NEXT_PUBLIC_APP_URL` (used in `.env.local`, create-order, webhook, onboarding)
- `NEXT_PUBLIC_SITE_URL` (used only in callback-server GET handler)
- Port inconsistency: 3000 (stale example) vs 3003 (actual)

**Fix**: Standardize on ONE variable (`NEXT_PUBLIC_APP_URL`), fix the callback-server reference, remove stale example.

### RC2: Two Onboarding Wizards
**Root**: The `(onboarding)/page.tsx` was the original comprehensive wizard. Later, a simpler `onboarding/page.tsx` was added (perhaps for a different flow). Route group `(onboarding)` resolves to the same URL as plain `onboarding`.

**Fix**: Remove one. The 4-step wizard `(onboarding)/page.tsx` is more complete and should be kept. The 2-step one can be repurposed as an "Add Store" flow under a different route (e.g., `/dashboard/stores/new`).

### RC3: Double Tenant Creation
**Root**: The auth callback-server creates a tenant + user + subscription when a new user signs up. Then the onboarding API also creates a tenant + user + subscription + store + categories + personas. Both paths can fire for the same user.

**Fix**: Callback-server should only create the minimal user record. Onboarding API should handle the full tenant setup. Or: callback-server creates everything and onboarding only creates the store.

### RC4: Auth Provider Flags Not Set
**Root**: The signup page conditionally renders auth methods based on `NEXT_PUBLIC_ENABLE_*` env vars. If these are undefined (not set), the equality check `=== 'true'` fails, and **no auth method renders**.

**Fix**: Either set these in `.env.local` and `.env.local.example`, or change the logic to default to `true` when undefined (i.e., `process.env.NEXT_PUBLIC_ENABLE_EMAIL_AUTH !== 'false'`).

### RC5: In-Memory Rate Limiting on Serverless
**Root**: Rate limiter uses `Map` in module memory. On Vercel, each serverless function invocation gets a fresh memory space. Rate limits never accumulate.

**Fix**: Use an external store (Redis via Upstash, Vercel KV, or Supabase table) for rate limiting.

## 5. Fix Recommendations (Priority Order)

### Immediate (Before Any Testing)

1. **Fix B3 — Auth provider flags**: Change signup page logic to default to enabled when env var is undefined:
   ```ts
   const ENABLE_EMAIL_AUTH = process.env.NEXT_PUBLIC_ENABLE_EMAIL_AUTH !== 'false'
   const ENABLE_GOOGLE_AUTH = process.env.NEXT_PUBLIC_ENABLE_GOOGLE_AUTH === 'true'
   const ENABLE_PHONE_AUTH = process.env.NEXT_PUBLIC_ENABLE_PHONE_AUTH === 'true'
   ```
   Email should be default-on. Google/Phone should require explicit opt-in.

2. **Fix B1 — Env var name**: Change `callback-server/route.ts:92` from `NEXT_PUBLIC_SITE_URL` to `NEXT_PUBLIC_APP_URL`.

3. **Fix B2 — Duplicate onboarding**: Delete `src/app/onboarding/page.tsx` (the 2-step one). Keep `(onboarding)/page.tsx` (the 4-step one). Move the "add store" functionality to `/dashboard/stores` page.

4. **Fix B4/B5 — Production URLs**: Set `NEXT_PUBLIC_APP_URL` and PhonePe callback/redirect URLs to the production domain in Vercel environment variables.

### Short-Term (Before Production Launch)

5. **Fix C5 — Login doesn't create user in DB**: After successful `signInWithPassword`, call `/api/auth/callback-server` to ensure user exists in Prisma DB.

6. **Fix C7 — Double tenant creation**: Make callback-server idempotent — check if tenant already exists before creating. Use the user's email as the dedup key.

7. **Fix C1/C2 — Trial inconsistency**: Standardize on 14-day trial, `STARTER` plan (matching the Launch tier the user selected).

8. **Fix C3 — Rate limiter**: Replace in-memory rate limiter with Vercel KV or Upstash Redis.

9. **Fix C8 — Hardcoded VPA**: Remove fallback `ezvento@ybl`, show error if store VPA not configured.

10. **Fix H1 — Dead link**: Change `/dashboard/products` to `/dashboard/inventory`.

11. **Fix H7 — Middleware contradiction**: Remove `/onboarding` from public routes list (it requires auth per the redirect logic).

### Medium-Term (Quality & Reliability)

12. **Fix M2 — Plan data duplication**: Create a shared `plans.ts` config file, import everywhere.

13. **Fix M5 — Login session race**: Add `await supabase.auth.getSession()` after `signInWithPassword` before redirecting.

14. **Fix M6 — CSRF protection**: Add origin header check on state-changing API routes.

15. **Fix L1 — Middleware deprecation**: Migrate from `middleware.ts` to the new Next.js 16 `proxy` convention.

## 7. QA Strategy Going Forward

### Test Pyramid
| Layer | Tool | Coverage | Frequency |
|-------|------|----------|-----------|
| Unit | Vitest | Business logic, plan limits, API helpers | Every commit |
| Integration | Vitest + MSW | API routes with mocked DB | Every PR |
| E2E | Playwright | Critical user journeys (signup → POS) | Nightly + pre-deploy |
| Visual | Chromatic/Percy | Landing page, auth pages, dashboard | Weekly |
| Load | k6/Vercel Speed Insights | POS billing under load | Pre-release |
| Security | OWASP ZAP + manual | Auth, payments, RLS | Monthly |

### Critical Test Scenarios (Must Pass Before Launch)
1. New user can sign up with email and complete onboarding
2. New user can sign up with Google OAuth and complete onboarding
3. Login works for existing users after email verification
4. Subscription payment via Dodo succeeds and activates account
5. Onboarding creates correct tenant, store, categories, and personas
6. POS can create a bill with cash payment
7. POS offline mode works (create bill, sync when online)
8. Role-based access control works (billing operator can't access settings)
9. RLS prevents cross-tenant data access
10. Rate limiting prevents brute force auth attempts

### Monitoring (Post-Launch)
- **Auth success/failure rate** — track signup → email verify → login conversion
- **Payment funnel** — signup → payment page → checkout → success conversion
- **Onboarding completion** — payment success → store created → first bill created
- **Error rates** — 5xx responses on auth and payment API routes
- **Session issues** — track `session_error` and `no_session` callback errors

### Supabase Dashboard Configuration Required
Before production, verify these settings in Supabase Dashboard:
1. **Site URL**: Set to production domain (not localhost)
2. **Redirect URLs**: Add `https://yourdomain.com/auth/callback` and `https://yourdomain.com/api/auth/callback-server`
3. **Email templates**: Verify confirmation email links use production URL
4. **Google OAuth**: Add production callback URL to Google Cloud Console
5. **Phone OTP**: Enable SMS provider for Indian numbers
6. **RLS**: Verify `apply-rls.ts` has been run against production database
